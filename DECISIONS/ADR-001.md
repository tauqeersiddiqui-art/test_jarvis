# ADR-001: Bounded Provider Failover Chain for Coding/Text Completion

Status: Accepted (already implemented — this ADR documents an existing decision,
retroactively, per the documentation architecture initiative of 2026-07-13/14).

## Problem

Coding/text completion calls (planner/writer in `actions/dev_agent.py`, investigation
synthesis in `actions/investigate.py`, etc.) need a model backend. Relying on a single
hardcoded provider means any transient failure (rate limit, capacity, timeout,
temporarily unsupported model) fails the entire coding operation, even though the
failure has nothing to do with the request itself.

## Alternatives Considered

1. **Single hardcoded provider, no failover.** Simplest, but any transient provider
   issue blocks all coding/text work until it clears.
2. **Retry the same provider with backoff.** Handles rate limits somewhat, but does
   nothing for a provider outage or a genuinely unsupported model.
3. **Bounded, ordered failover across multiple configured providers, with error
   classification.** Tries the next configured provider only on a *recoverable*
   error; fails fast on anything else so a real bug isn't silently masked.

## Decision

Implement option 3: `core/ai_provider.py`'s `complete_with_failover()` builds an
ordered chain (OpenAI-compatible gateway primary if fully configured via
`LLM_API_KEY`/`LLM_BASE_URL`/`LLM_MODEL`, Gemini fallback if `GEMINI_API_KEY` is set),
classifies each failure (`rate_limit`, `payment_required`, `capacity`, `unavailable`,
`model_unavailable` vs. `unknown`), and only advances the chain on a recoverable
category. The chain is bounded by the number of configured providers — no infinite
retries.

Model selection is a separate concern from provider selection: an explicit `model`
argument is only ever forced onto a Gemini provider, never onto an OpenAI-compatible
gateway, which keeps using its own configured model (which may itself be an
omni-routing "auto" selection).

This chain is explicitly scoped to coding/text completion only. The Gemini Live
real-time voice session (`main.py`) and the vision session
(`actions/screen_processor.py`) are deliberately kept outside this abstraction — see
`ARCHITECTURE.md` §5 and `DECISIONS/ADR-006.md`.

## Reasoning

- A real bug or bad request must not be silently masked by switching providers —
  hence the recoverable/unknown split, not a blanket try-next-on-any-error policy.
- Provider independence (no code outside `core/ai_provider.py` should know which
  backend is active) matches `PRODUCT_VISION.md`'s Technology Principle "Replaceable
  components."
- Bounding the chain avoids a retry storm against a genuinely down set of providers.

## Consequences

- Every future coding/text-completion consumer must call
  `complete_with_failover()` rather than constructing its own client — enforced by
  convention and code review, not by a runtime guard.
- Adding a new provider means adding a class implementing `AIProvider` and wiring it
  into `build_failover_chain()` — no changes needed in any calling module.
- The Gemini Live voice/vision sessions remain a deliberate, documented exception to
  "everything goes through the provider abstraction" — this is a known architectural
  inconsistency, tracked in `DECISIONS/ADR-006.md`, not something this ADR resolves.
