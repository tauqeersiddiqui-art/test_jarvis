# ADR-005: Adapt, Not Adopt Wholesale, Ideas from the Historical mini_agent Reference

Status: Accepted (already implemented — this ADR documents an existing decision,
retroactively).

## Problem

The historical reference project `D:\All Bots\mini_agent` (JARVIS OS) had already
built conceptually similar capabilities before this repository did — a searchable
engineering-memory system (`missions/engineering_memory.py`) and a dependency-graph
impact-analysis primitive (`self_engineering.py`). When building
`core/engineering_memory.py` and `actions/impact_analysis.py` in this repository, a
choice had to be made about how much of that prior design to reuse.

## Alternatives Considered

1. **Integrate directly** — depend on or vendor `mini_agent`'s modules as-is.
2. **Reference only** — read for inspiration, write something structurally unrelated.
3. **Adapt** — take the useful design idea, rebuild it as a Mark-native primitive
   using this repository's own conventions, with no shared runtime or dependency.

## Decision

Implement option 3 for both modules. `core/engineering_memory.py`'s own docstring
states it "adapts the useful idea from mini_agent's missions/engineering_memory.py...
into a Mark-native primitive that reuses core.coding_task's persistence conventions...
no new execution engine, no LLM calls of its own." `actions/impact_analysis.py`
likewise "adapts the useful part of mini_agent's self_engineering.py... into a small
Mark-native primitive, strictly scoped to one project_root via core.workspace's
existing boundary primitives, and reusing actions.codebase_search rather than
reimplementing file discovery/search."

Per the classification scheme in `PRODUCT_VISION.md` Part 6: **Integrate** was
rejected (option 1) — `mini_agent` is a permanently frozen historical reference
project (see `PRODUCT_VISION.md`), never an active dependency of this repository.
**Reference** (option 2) was rejected as leaving value on the table — the underlying
design ideas (bounded searchable outcome records; a dependency graph + bounded impact
report) were sound and worth reusing structurally. **Adapt** (option 3) was chosen.

## Reasoning

- `mini_agent` is architecturally a different system (a plugin/Kernel-based
  architecture with its own persistence layer) — directly depending on its modules
  would import an entire foreign architecture into this repository for a small piece
  of functionality.
- Rebuilding as a Mark-native primitive means the result reuses *this* repository's
  existing conventions (`core.workspace`'s boundary, `core.coding_task`'s persistence
  pattern, `actions.codebase_search`'s search) rather than introducing a second,
  parallel set of conventions.
- This keeps `mini_agent` genuinely frozen — no live coupling, no version-drift risk,
  no shared runtime to keep in sync.

## Consequences

- Future work that finds a good idea in `mini_agent`'s frozen documentation (its
  Cognitive OS design, its mission-engine pipeline, its plugin architecture) should
  default to this same "adapt" pattern rather than importing or vendoring `mini_agent`
  code directly, unless a future ADR explicitly revisits this.
- This is the concrete precedent behind `PRODUCT_VISION.md` Part 6's "Adapt" row in
  the Integrate/Adapt/Reference/Reject classification table.
