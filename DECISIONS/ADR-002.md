# ADR-002: Single-Slot, Single-JSON-File State Persistence

Status: Accepted (already implemented — this ADR documents an existing decision,
retroactively).

## Problem

`core/coding_task.py`, `core/engineering_memory.py`, and `core/pending_action.py` each
need to persist small amounts of state across conversation turns and app restarts:
the current coding project, past engineering outcomes, and an in-flight
slot-filling action, respectively. This state needs to survive a crash without
corruption and without requiring a database dependency.

## Alternatives Considered

1. **A real embedded database (SQLite).** Robust, but adds a dependency and schema
   migration surface for what is, per module, a very small amount of state in a
   single-user, single-session application.
2. **Multiple JSON files per record, one file per task/record.** Avoids read/write
   contention on one file, but doesn't match the actual usage pattern (only one
   active coding task and a bounded, small number of engineering-memory records are
   ever relevant at once) and adds directory-scanning overhead for no real benefit at
   this scale.
3. **Single small JSON file per concern, single-slot where only one active record
   makes sense, atomic writes.**

## Decision

Implement option 3. Each of `core/coding_task.py` (single active task),
`core/engineering_memory.py` (a bounded list of records, pruned by value/recency), and
`core/pending_action.py` (single in-flight pending action) persists to one small JSON
file under `config/state/` (gitignored), written atomically via a temp file +
`os.replace()` (atomic on both POSIX and Windows when source and destination share a
volume).

## Reasoning

- This application is single-user/single-session (one active Gemini Live conversation
  at a time) — one active coding task and one in-flight pending action is all that is
  ever needed; a single-slot design matches reality rather than generalizing for
  concurrency that doesn't exist.
- `core/engineering_memory.py` bounds retention explicitly
  (`MAX_RECORDS_PER_PROJECT`, `MAX_TOTAL_RECORDS`) rather than growing unbounded,
  keeping the "database-free" approach viable long-term.
- A missing or corrupt state file must never crash the assistant — every loader
  (`load_active_task()`, `_load_all()`) fails safe, returning `None`/`[]` rather than
  raising.
- Atomic writes prevent a crash mid-write from corrupting the previous good state.

## Consequences

- Adding a genuinely concurrent/multi-project capability (e.g. `PRODUCT_VISION.md`
  Track I, Project Digital Twin, or a multi-project Enterprise Edition) cannot simply
  widen these single-slot files — it needs its own, separately-scoped persistence
  design (tracked in `ROADMAP.md` Phase 7), not a retrofit of this pattern.
- Every new Core Services module needing small persisted state should default to this
  same pattern (single JSON file, gitignored, atomic write, fail-safe load) unless a
  specific reason requires otherwise — consistency here makes the whole `config/state/`
  directory predictable to reason about.
