# ADR-006: Vision's Second Gemini Live Session Is a Known Exception, Not a Pattern to Extend

Status: Accepted as a documented, tracked exception. Not resolved by this ADR — no
source code change is proposed or made here.

## Problem

`actions/screen_processor.py` opens its own, independent Gemini Live audio session to
describe screen/camera captures, separate from both `main.py`'s primary voice session
and `core/ai_provider.py`'s coding/text failover chain (ADR-001). This is a pre-
existing characteristic of the running system, discovered during the documentation
architecture work of 2026-07-13/14, not a new decision being made now. It matters now
because `PRODUCT_VISION.md` Track D (Yoga Coach: real-time pose estimation) and
Track E (Teacher: attention/eye-direction tracking) both propose *additional*
camera-based analysis, which naturally raises the question of whether each should get
its own session too.

## Alternatives Considered

1. **Leave as-is, and let each new camera-analysis capability (Track D, Track E) open
   its own additional independent session.** Simplest per-feature, but multiplies an
   already-flagged inconsistency and increases the risk of the same
   echo-loop/self-triggering class of bug that `screen_processor.py` already had to
   build cooldown/busy-flag guards for.
2. **Leave as-is for the existing module, but require future camera-analysis work to
   route through the existing vision session rather than opening a new one.**
3. **Refactor `screen_processor.py` now to unify with `core/ai_provider.py`'s
   abstraction.** Would resolve the inconsistency immediately, but is a source-code
   change outside the scope of a documentation-architecture pass, and risks
   destabilizing a working, tested capability without a dedicated proposal.

## Decision

Adopt option 2 for now: `actions/screen_processor.py`'s existing second session is
left untouched (no source code changed by this documentation work, per explicit
instruction), but it is formally flagged in `ROADMAP.md` (Phase 1, "Known issues"),
`PRODUCT_VISION.md` (Part 2 §5), and `MODULES/Vision.md` as an exception to the
"Gemini Live stays singular and separate" architectural invariant
(`ARCHITECTURE.md` §5). Any future Track D or Track E implementation must route
camera-based analysis through this existing session rather than opening a third
independent one. A future ADR should revisit option 3 (unification) if and when
Track D or Track E work is actually proposed and scoped.

## Reasoning

- Documentation work should record reality accurately, including known
  inconsistencies, rather than silently omitting them — this ADR exists so the
  inconsistency isn't rediscovered from scratch by whoever picks up Track D or E.
- Multiplying independent sessions multiplies the exact class of bug
  (`screen_processor.py`'s own cooldown/echo-guard logic exists specifically to
  handle this session picking up its own narrated output via the microphone) rather
  than fixing it.
- Actually unifying the sessions is a real code change with real risk to a working
  capability, and belongs to its own explicitly-scoped, tested proposal — not a
  documentation pass.

## Consequences

- `ROADMAP.md` Phase 6 (Domain Expert Modes) explicitly carries this constraint
  forward for Track D and Track E.
- If a future session decides to unify the vision session with
  `core/ai_provider.py`'s abstraction, that work should open a new ADR (ADR-007 or
  later) rather than silently amending this one — this ADR's decision (defer,
  contain, document) stands as the historical record of what was decided and why,
  even after a later resolution.
