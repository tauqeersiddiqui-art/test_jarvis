# ADR-008: Loop Detection Lives in the Orchestrator, Not in dev_agent.py's Pipelines

Status: Accepted (implemented 2026-07-14, alongside the Loop Detection capability).

## Problem

`actions/dev_agent.py`'s fix/feature pipelines can, in principle, keep retrying the
same unsuccessful approach across multiple separate `dev_agent()` calls (each one a
fresh user request to "fix it" or "try again"), each spending at least one model call
without any deterministic check for whether real progress is being made. A way was
needed to detect this and stop before the next model call, without adding a new
decision system inside the pipelines themselves or duplicating data those pipelines
already produce.

## Alternatives Considered

1. **Instrument the loop check inside `actions/dev_agent.py`'s `_run_fix_loop()` /
   `_run_incremental_feature_change()`.** Would require touching working,
   already-tested pipeline internals — explicitly out of scope for this capability
   ("no redesign, no architecture rewrite").
2. **Give `core/loop_detector.py` its own persistence, recording its own history of
   attempts.** Would duplicate data `core/execution_ledger.py` and
   `core/engineering_memory.py` already record — explicitly disallowed
   ("do not duplicate logic").
3. **Have `core/coding_orchestrator.py` call `core/loop_detector.py` before routing a
   `CONTINUE_FIX`/`CONTINUE_FEATURE` request to the existing pipeline**, using only
   already-persisted, already-isolated-in-tests data from
   `core/execution_ledger.py` and `core/engineering_memory.py`.

## Decision

Implement option 3. `core/coding_orchestrator.py`'s `decide()` calls
`core/loop_detector.py`'s `check_for_loop(active.task_id)` immediately after
confirming there's an active task being continued (before the fix-vs-feature
classification), and — only when a loop is detected — returns a new
`Route.LOOP_DETECTED` instead of `CONTINUE_FIX`/`CONTINUE_FEATURE`. No pipeline
function runs, so no further model call is spent. `actions/dev_agent.py`'s
`dev_agent()` handles `LOOP_DETECTED` exactly like `MISSING_DESCRIPTION`/
`NEEDS_CLARIFICATION` — return the message, call no pipeline.

`core/loop_detector.py` itself has no persistence of its own — it is a pure function
of `core/execution_ledger.py`'s `entries_for_task()` and
`core/engineering_memory.py`'s `records_for_task()` (a new, small, additive read
accessor on that module, mirroring the ledger's own `entries_for_task()`).

## Reasoning

- The orchestrator is already the single place that decides "what kind of coding task
  is this, where should it go" (`DECISIONS/ADR-006.md`'s sibling extraction,
  `core/coding_orchestrator.py` itself) — adding "should this even be routed to the
  pipeline at all" to the same decision point is a natural extension of that role, not
  a new architectural layer.
- Reusing `execution_ledger`/`engineering_memory` as the sole data sources means loop
  detection required zero new instrumentation inside `dev_agent.py`'s pipelines, and
  zero new persistence — exactly the "small implementation, no duplicate logic"
  constraint this capability was given.
- Returning a routing decision (`LOOP_DETECTED`) rather than raising an exception or
  silently declining keeps the orchestrator's existing contract: every call to
  `decide()` returns a `RoutingDecision` the caller acts on, never throws for a normal
  business outcome.
- Every repetition-based check (`REPEATED_ROUTING_DECISION`, `REPEATED_FILES_TOUCHED`,
  `REPEATED_FINGERPRINT`, `REPEATED_ERROR_SIGNATURE`) is explicitly guarded against
  false positives on legitimate repeated success (e.g. three consecutive successful
  feature additions using the same route) — only a repetition with **zero** successful
  outcomes in the window counts as a loop signal.

## Consequences

- Loop detection only ever applies to `CONTINUE_FIX`/`CONTINUE_FEATURE` requests on an
  *existing* task — `NEW_PROJECT` requests are never checked, since there is no prior
  history to repeat yet.
- If a future capability wants loop detection to also gate the initial build path (a
  fresh project whose planning loops internally), that is a separate, explicitly-scoped
  extension — not implied by this ADR.
- `core/loop_detector.py`'s `DEFAULT_WINDOW = 3` is a simple, fixed threshold, not
  configurable per-task or adaptive. If real usage shows this threshold is wrong in
  either direction, that is a follow-up tuning proposal, not a redesign.
