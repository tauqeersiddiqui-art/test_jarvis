# ADR-007: Execution Ledger Result Classification Derives from CodingTask.Status, Not a New Signal

Status: Accepted (implemented 2026-07-14, alongside the Execution Ledger capability).

## Problem

The Execution Ledger (`core/execution_ledger.py`) needs to record a `result` for each
routed coding operation — success, failure, or rollback — for every `dev_agent()` call.
`CodingTask.Status` (`core/coding_task.py`) only has `ACTIVE`/`COMPLETED`/`FAILED`/
`ARCHIVED` — no `ROLLED_BACK` value. A way was needed to derive a 3-way ledger result
(`success`/`failure`/`rollback`) from the 2-way completion signal that already exists,
without adding a new instrumentation point deep inside `actions/dev_agent.py`'s
private fix/feature pipelines.

## Alternatives Considered

1. **Add a new `Status.ROLLED_BACK` value to `core/coding_task.py`.** Would require
   changing an existing, working state machine used throughout `dev_agent.py` and its
   tests — exactly the kind of architecture change this capability was told not to
   make.
2. **Have the ledger read `core/engineering_memory.py`'s per-attempt records
   (`em.OUTCOME_ROLLED_BACK`/`OUTCOME_FAILED`) to determine the true outcome.** Would
   require reaching into `engineering_memory.py`'s private `_load_all()` and
   re-deriving a top-level result from per-attempt records — indirect, and risks
   duplicating logic `engineering_memory.py` already owns for a different purpose
   (informing future attempts, not top-level observability).
3. **Derive the ledger's `result` directly from `task.status` immediately after the
   pipeline call returns**, using the existing, already-verified invariant that every
   `FAILED` status in this codebase's fix/feature pipelines is reached only after a
   full rollback to the last good state.

## Decision

Implement option 3. `actions/dev_agent.py`'s `_log_ledger_entry()` maps:
- `CodingTask.Status.COMPLETED` → `Result.SUCCESS`
- `CodingTask.Status.FAILED` → `Result.ROLLBACK`
- anything else (task still `ACTIVE`, e.g. a test double that didn't transition
  status) → `Result.FAILURE`, logged as an unexpected state for engineering
  visibility rather than silently assumed to be a success.

## Reasoning

- Every code path in `actions/dev_agent.py` that reaches `ct.mark_failed()` does so
  only after `_rollback_snapshot()` has already restored the project directory to its
  last good state — confirmed directly in this codebase's own tests
  (`test_build_project_rolls_back_on_unchanged_error`,
  `test_incremental_change_worsening_fix_attempt_still_rolls_back`, both assert file
  content equals the original/rolled-back baseline exactly when status becomes
  `FAILED`). So `FAILED` in this codebase's design already *means* "rolled back," not
  merely "gave up" — mapping it to `Result.ROLLBACK` is an accurate label, not an
  assumption.
- This requires zero new instrumentation inside `dev_agent.py`'s private pipelines
  (`_build_project`, `_continue_fix_loop_for_task`, `_run_incremental_feature_change`,
  `_run_fix_loop`) — the ledger reads a signal (`task.status`) those pipelines were
  already producing for `core/coding_task.py`'s own purposes.
- `Result.FAILURE` is reserved for a state this design doesn't expect to reach in
  normal operation (task still `ACTIVE` after the pipeline returned) — this makes the
  ledger a genuinely useful anomaly signal rather than a value that's never actually
  used.

## Consequences

- If a future change to `actions/dev_agent.py` ever introduces a `FAILED` path that
  does *not* roll back (e.g. a partial, irrecoverable failure), this ADR's mapping
  would silently mislabel it as `ROLLBACK` — any such change must either preserve the
  rollback-before-`FAILED` invariant or revisit this ADR.
- If `core/coding_task.py` ever gains a genuine `Status.ROLLED_BACK` value (option 1,
  revisited later), `_log_ledger_entry()`'s mapping should be simplified to read it
  directly instead of inferring rollback from `FAILED`.
