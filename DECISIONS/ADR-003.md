# ADR-003: Resolve-Then-Contain Workspace Security Boundary

Status: Accepted (already implemented — this ADR documents an existing decision,
retroactively).

## Problem

Any module that lets JARVIS search, read, or reason about files based on a user- or
model-supplied path needs a way to guarantee that path can never escape the active
workspace directory — otherwise a crafted or accidental `../../` path (or a symlink)
could expose files far outside the intended project.

## Alternatives Considered

1. **String-prefix check on the unresolved path.** Cheap, but defeated trivially by
   `..` segments or symlinks — checking containment before resolving is exactly the
   kind of check that a `../../etc/passwd`-style path slips through.
2. **Deny-list of dangerous path patterns.** Fragile — an incomplete deny-list is a
   security bug waiting to be found; also doesn't handle symlinks that point outside
   the workspace.
3. **Resolve fully first (symlinks, `..`, relative segments), then check containment
   against the resolved workspace root.**

## Decision

Implement option 3 in `core/workspace.py`'s `resolve_in_workspace()`: the candidate
path is resolved to an absolute, symlink-free path via `Path.resolve()`, and *only
then* checked with `relative_to()` against the resolved workspace root. Any path that
fails this containment check raises `PathEscapeError` rather than silently truncating
or ignoring it.

A parallel, separate guard (`is_sensitive()`) blocks *content* access to a fixed set
of sensitive filenames and extensions (`.env` variants, API keys, credentials,
`.netrc`, private keys, certificates) — independent of whether the path is inside the
workspace, since a sensitive file can legitimately live inside the workspace root.

## Reasoning

- Resolving first, then checking, is what actually defeats traversal — this is
  explicitly called out in the module's own docstring as the reason for this
  ordering, not an incidental implementation detail.
- Path containment and content sensitivity are orthogonal concerns: a file can be
  inside the workspace (so its *path* is fine to list) while its *content* must still
  never be read or sent to an AI provider. Keeping these as two separate checks
  (`resolve_in_workspace()` vs. `is_sensitive()`) avoids conflating "where" with
  "what."
- This mirrors the pattern already used elsewhere in the codebase
  (`actions/shell_exec.py`'s workspace-escape guard, `actions/git_control.py`'s
  subprocess-as-args-list-never-shell=True pattern) rather than inventing a third
  approach.

## Consequences

- Every module that resolves a file path for search/read/write must call through
  `core/workspace.py` rather than resolving paths independently — this is the "one
  workspace boundary" invariant recorded in `ARCHITECTURE.md` §5.
- `SENSITIVE_NAMES`/`SENSITIVE_EXTENSIONS` are a fixed list; extending them (e.g. for
  a new secrets-file convention introduced by a future Track) requires an explicit
  edit to `core/workspace.py`, not a per-caller workaround.
- Any future Expert Mode (Track C) that reads user files outside the coding-project
  workspace (e.g. Track F's property documents, Track N's government forms) will need
  its own explicit workspace-boundary decision — this ADR's pattern should be reused,
  but the specific sensitive-file list may need to be extended for those domains.
