# ADR-004: Reject Blind UI Automation as a Pattern for New Cross-App Control

Status: Accepted. `actions/send_message.py` remains in the codebase as existing,
working functionality â€” this ADR governs *future* work, it does not deprecate the
existing module.

## Problem

`actions/send_message.py` automates messaging apps (WhatsApp/Signal/Discord/Instagram)
via `pyautogui` â€” blind coordinate and keystroke replay against whatever is currently
on screen, with no semantic understanding of the target application's UI. This works
today, but any future capability that needs to control a third-party desktop
application (Track E's student messaging touches, Track N's government-portal form
filling) will face the same design choice: automate blindly, or automate against
explicit, addressable UI elements.

## Alternatives Considered

1. **Extend the blind coordinate/keystroke pattern to new surfaces.** Fastest to
   write, matches existing precedent in this codebase.
2. **Prefer official APIs where they exist** (e.g. a messaging platform's Bot API, a
   government portal's published API if one exists).
3. **Where no official API exists, use Playwright (or an equivalent) with explicit
   selectors** â€” addressing UI elements semantically rather than by screen
   coordinates.

## Decision

For any *new* cross-application automation surface, prefer option 2, falling back to
option 3. Do not extend option 1's blind-coordinate pattern to new applications or
domains. `actions/send_message.py` is not retroactively rewritten by this decision â€”
it continues to function as-is â€” but it is not the template for what comes next.

`actions/browser_control.py` already demonstrates the preferred alternative for
browser-based targets.

## Reasoning

- Blind coordinate/keystroke automation is fragile by construction: it breaks on any
  UI change (a redesign, a moved button, a different screen resolution, a changed
  window size) with no error signal beyond "it silently clicked the wrong thing."
- `actions/game_updater.py` exhibits the same pattern (`pywinauto`, `winreg`,
  `schtasks`, screen-pixel dialog automation) for a different, niche purpose â€” it is
  not being extended either, for the same reason.
- Track E (Teacher) and Track N (India Workflow Assistant) both involve
  higher-stakes, higher-consequence actions (student communication, government
  document submission) than the existing messaging use case â€” the cost of a silent
  misclick is proportionally higher, making fragility less acceptable, not more.
- This decision is consistent with `PRODUCT_VISION.md`'s Technology Principle
  "Security first" / "Human remains in control" â€” an automation approach that can
  silently act on the wrong on-screen element undermines both.

## Consequences

- Any proposal for Track E's or Track N's automation surfaces must identify an
  official API first (per the Open Source Strategy / integration-classification
  discipline in `PRODUCT_VISION.md` Part 6) before defaulting to Playwright-with-
  explicit-selectors, and must not default to blind `pyautogui`-style automation.
- `actions/send_message.py` and `actions/game_updater.py` remain flagged in
  `ROADMAP.md` (Phase 0) as "complete, but not a pattern to extend" rather than being
  removed or rewritten â€” no source code change resulted from this ADR.
